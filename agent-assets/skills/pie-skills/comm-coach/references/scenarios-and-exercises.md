# 场景示例与练习库

本文件是 comm-coach Phase 3（真实场景）和 Phase 4（下一步练习）的**灵感库**。

**重要**：这些示例是**灵感参考，不是直接输出**。实际输出应该基于**用户当前会话的真实问题**生成场景和练习，才有针对性。这些示例只是展示"好的场景和练习长什么样"。

---

## 场景示例库（Phase 3 参考）

### 场景：复杂代码重构（AI 沟通）

**❌ 坏的 prompt**：
> "帮我重构这个组件，代码太乱了。"
> [粘贴 300 行代码]

**问题**：
- 模糊表达（"太乱"主观）
- 无角色设定
- 无约束声明（什么不能改）
- 无输出规范

**✅ 用 RACE + CoT 改写**：
> **R**: 你是一个 React 性能优化专家。
> **A**: 重构 `components/ProductList.jsx`，问题是渲染 1000+ 商品时卡顿。
> **C**: 约束：(1) 不能改 props API（外部调用多），(2) 不引第三方状态管理库，(3) 保留筛选排序功能。
> **E**: 先分析性能瓶颈 → 给重构方案（附为什么这样改）→ 重构后代码 → 性能验证方法。

### 场景：学习新技术概念（AI 沟通）

**❌ 坏的 prompt**：
> "给我讲讲 Kubernetes"

**问题**：没说到什么深度、没说已有基础、没说期望格式

**✅ 用 CO-STAR 改写**：
> **C**: 我是有 3 年 Docker 经验的后端开发者，想转向云原生架构。
> **O**: 理解 K8s 核心概念，能部署简单应用。
> **S**: 类比 + 实际例子，避免纯理论。
> **T**: 轻松友好。
> **A**: 唯一学习者，不考虑其他人。
> **R**: 5 个概念模块，每模块 200 字，附一个动手练习。

---

### 场景：调试疑难问题（AI 沟通）

**❌ 坏的 prompt**：
> "这个报错怎么解决？" + 一段 stack trace

**问题**：没说本地能否复现、没说已尝试过什么、没说环境

**✅ 改写**：
> **问题**：生产环境每天约 50 次 `NullPointerException`，本地无法复现。
> **已尝试**：(1) 本地高并发模拟 - 无法复现，(2) 检查输入验证 - 看起来无问题，(3) 加日志 - 等新数据。
> **环境差异**：生产 AWS Lambda，本地 Docker。
> **期望**：按可能性排序根因方向，每个方向给验证方法。

### 场景：向甲方汇报延期（人际沟通）

**❌ 坏的**：
> "项目可能会延期一点，支付接口有些技术问题比较麻烦，我需要再研究一下。"

**问题**："一点""比较麻烦""可能" 全模糊，缺具体数字、原因、方案

**✅ 用 SCQA + Pyramid**：
> 项目已完成 85%，核心功能在测试环境（**S**）。
> 支付接口的银行级安全认证要补 3 个合规文档，审核 3-5 工作日（**C**）。
> 这意味着交付延至下周五（**Q**）。两个方案：
> **方案 A（推荐）**：本周五交付可测试版，下周五上线真实支付。
> **方案 B**：下周五一次性交付完整版。
> 您倾向哪个？（**A**）

### 场景：团队技术讨论（人际沟通）

**❌ 坏的**：
> "我觉得没必要换，现在的够用了。"

**问题**：无具体理由、不承认对方观点合理性、易引发对立

**✅ 用 PREP**：
> **P**: 建议暂时不引入新库。
> **R**: 三个考虑：学习成本、迁移风险、当前库满足需求。
> **E**: 上次引入 Redux Toolkit，团队花 2 周熟悉，期间 3 个生产 bug。
> **P**: 所以建议等现有方案不够用时再考虑。但我愿意听不同意见。

## 练习库（Phase 4 参考）

### 练习 1：RACE 速写（3 分钟）

**何时推荐**：用户有角色定义 / 约束声明问题

**模板**：
```
R (Role): 你是一个 ________
A (Action): 请帮我 ________
C (Context): 背景 ________，约束 ________
E (Expectation): 输出格式 ________
```

**预期效果**：AI 理解准确率从 60% → 90%+，减少往返澄清。

---

### 练习 2：8 维度自检清单（每次 prompt 前）

**何时推荐**：用户有多个低分维度

**清单**：
```
□ 角色 Role: 是否说了"你是一个___专家"？
□ 上下文 Context: 是否提供足够背景？
□ 约束 Constraints: 是否说了"不能___"？
□ 输出规范 Output: 是否说了期望格式？
□ 模糊程度 Vague: 有"改一下""优化"等空泛词吗？
□ 需求框架 Why/How/Who: 说清楚了吗？
□ 元沟通 Meta: 是否需要先确认 AI 理解？
□ 反馈 Feedback: 上一轮是否给 AI 反馈？
```

**预期效果**：养成结构化提问习惯。

---

### 练习 3：反馈循环（每次 AI 回复后）

**何时推荐**：用户的反馈维度低分

**模板**：
```
AI 回复后问自己：
1. AI 是否理解了我的真实需求？(是/否/部分)
2. 如果否，哪里偏差？
3. 下次如何调整？

然后对 AI 说：
"你理解了 ___，但我实际想要 ___，请重新 ___"
```

**预期效果**：从"将就用 AI 回复" → "引导 AI 给精确答案"。

---

### 练习 4：LLM-as-a-Judge（自我评估）

**何时推荐**：用户想长期提升，想要"第三方视角"

**模板**：
```
发送真正的 prompt 前，先对 AI 说：

"请评估下面这个 prompt 的质量（8 维度打 1-5 分）：
1. 角色定义 2. 上下文 3. 约束 4. 输出规范
5. 模糊程度 6. 需求框架 7. 元沟通 8. 反馈循环

prompt: '[你的 prompt]'

请给打分表格和具体改进建议。"
```

**预期效果**：养成发送前自检习惯，内化评估标准。

---

## 挑练习的启发式

**不要每次 4 个全推**。根据 Phase 1 诊断的 Top 3-5 问题挑 1-2 个最对口的：

- 主问题是**约束/角色/输出**低分 → 练习 1（RACE 速写）
- 主问题是**多维度分散** → 练习 2（8 维度清单）
- 主问题是**反馈缺失 / 元沟通** → 练习 3（反馈循环）
- 用户想**系统性提升** / 是第 2 次以上使用 → 练习 4（LLM Judge）

第 N 次使用的用户（N ≥ 3）应该推新练习，不要重复之前的。
