# 沟通教练 Communication Coach

## 这个技能干什么

扫描用户最近的对话，找出沟通/表达里的 Top 3-5 个问题，推荐经过搜索验证的框架，生成针对性的改进场景和练习，**并把结果写入 vault 追踪长期进步**。

**核心原则**：
1. **基于证据** —— 引用用户**真实原话**（逐字 + turn 编号），不编造例子
2. **搜索验证** —— 推荐的框架必须有真实来源支撑（2 个以上来源交叉验证），不靠记忆凭空说
3. **针对性生成** —— 场景和练习从**本次会话的真实问题**出发，不用写死的 demo
4. **持续追踪** —— 每次分析写入 vault，对比历史进步

---

## 工作流（5 个阶段）

```
Phase 0: 读历史     → Phase 1: 诊断     → Phase 2: 推荐框架（含搜索验证）
                                                ↓
Phase 4: 持久化    ← Phase 3: 场景+练习 ←
```

---

### Phase 0 — 读历史（如果存在）

在开始分析前，先读 `COMM_COACH_DIR/index.json`（目录见下方 [Phase 4](#phase-4--持久化到-vault)）。

**如果存在**：
- 记住之前推荐过哪些框架（避免重复推荐给老用户）
- 记住上次低分维度（检查本次是否改善）

**如果不存在**：直接进 Phase 1，跳过历史回顾。

**为什么先读历史**：如果用户连续 3 次都在"约束声明"上被打 5/5 分，说明建议没落地，应该换种方式讲（比如给更具体的模板），而不是机械复读同一句建议。

---

### Phase 1 — 诊断（8 维度 → Top 3-5）

扫描用户的会话消息（范围见下方"扫描范围"），按 8 个维度给每个问题打 1-5 分（分数越高 = 问题越严重）：

| # | 维度 | 检测什么 |
|---|---|---|
| 1 | **模糊表达** | 空泛词："改一下"、"优化"、"有问题"、"差不多" |
| 2 | **上下文缺失** | 代词不明（"这个""那个"）、缺背景 |
| 3 | **需求框架** | 是否说了 Why（目标）/ How（期望）/ Who（场景） |
| 4 | **反馈循环** | 是否对 AI 回复给过反馈、是否确认理解正确 |
| 5 | **角色定义** | 是否给 AI 设了明确角色/专家身份 |
| 6 | **约束声明** | 是否说了"不能做什么"、边界条件 |
| 7 | **输出规范** | 是否说了期望格式（代码/列表/表格/步骤） |
| 8 | **元沟通** | 是否对 AI 理解进行确认/纠正/引导 |

### 扫描范围

- **会话长度 < 10 轮**：扫描全部
- **会话长度 10-50 轮**：扫描**最近 10 轮**用户消息
- **会话长度 > 50 轮**：扫描**最近 10 轮** + **前面任何明显有沟通问题的 5-8 条**（如果能识别的话）
- **会话长度 < 3 轮**：技能无法有效分析，输出提示后结束（见 [错误处理](#错误处理)）

### 输出规则

- **逐字引用**用户真实消息（不要改写），并标注 turn 编号（如 "消息 #5"）
- **合并语义相似的问题**：同一条消息的"模糊 + 上下文缺失"通常合并为一个综合问题
- **最终只输出 Top 3-5 个**，不要贪多

### Phase 1 输出格式

```markdown
## 🔍 对话分析 Conversation Analysis

### 8 维度评分

| 维度 | 发现 | 严重度 |
|---|---|---|
| 模糊表达 | 3 处 | 4/5 ⭐ |
| 上下文缺失 | 2 处 | 5/5 ⭐⭐ |
| 需求框架 | 1 处 | 3/5 |
| 反馈循环 | 0 处 | 0/5 |
| 角色定义 | 2 处 | 4/5 ⭐ |
| 约束声明 | 3 处 | 5/5 ⭐⭐ |
| 输出规范 | 1 处 | 2/5 |
| 元沟通 | 0 处 | 0/5 |

筛选后 Top 4：约束声明、上下文缺失、角色定义、模糊表达

---

### 1. 约束声明缺失（消息 #5）

**原文**（逐字引用）：
> "帮我重构这段代码"

**问题**：
- ❌ 没说什么不能改（API 兼容性？依赖？）
- ❌ 没说技术限制（性能？内存？）
- ❌ 没说范围边界

**改进版**：
> "请重构这段代码。约束：不改函数签名（外部调用多）、不加新依赖、保持 O(n) 时间复杂度"

**对应原则**：RACE 的 C (Context) 和 E (Expectation)

[继续 Top 2-5...]
```

### Phase 2 — 推荐框架（含搜索验证）

根据 Phase 1 的 Top 问题，从 [`references/frameworks.md`](references/frameworks.md) 的"按问题类型选框架"索引中挑 **2-3 个**最对口的。

**强制搜索验证**：
- 每个框架必须用 web 搜索工具（`web_search`、`mcp-omnisearch` 等当前环境可用的）验证**来源可靠性**
- **每个框架至少 2 个来源**交叉验证
- **必须输出 URL**（用户可以点开查验）
- 搜不到可靠来源 → 明确标注"未找到权威来源，以下为个人推断"，不要蒙

**为什么必须搜索**：如果不搜索就推荐，Claude 会出现幻觉——推荐一个听起来像真实但根本没人用的框架，用户按着错误框架学会吃大亏。搜索是这个技能**唯一的防线**。

**避免重复推荐**（根据 Phase 0 的历史数据）：
- 如果某框架已推荐过 ≥ 2 次、用户依然在对应维度低分 → 推荐**同类替代框架**（如 RACE 推过没用 → 推更具体的 TIDD-EC），并说明"上次推荐 RACE 效果不明显，这次换个角度"
- 如果首次推荐 → 正常流程

### Phase 2 输出格式

```markdown
## 📚 推荐方法论 Recommended Methodologies

### 1. CO-STAR Framework

**原理**：GovTech Singapore 2025 推广的结构化 prompt 框架。

**核心结构**：C-O-S-T-A-R（6 要素详见 [frameworks.md](references/frameworks.md#co-star)）

**为什么推你**：你的 Top 问题是**约束声明 + 上下文缺失**，CO-STAR 强制补齐 Context/Objective/Style/Audience 等要素。

**验证来源**：
- [URL 1] — GovTech Singapore 官方文档
- [URL 2] — [交叉验证来源]

**快速模板**：
```
C: [背景]
O: [目标]
S: [风格]
T: [语气]
A: [受众]
R: [格式]
```

[继续 2-3 个]
```

详细的框架资料、JSON schema、ID 规范 → [`references/frameworks.md`](references/frameworks.md)。

### Phase 3 — 场景 + 练习（针对性生成）

**不要照搬 demo**。根据 Phase 1 的 Top 问题**生成针对用户的场景**，最多 2 个 AI 沟通场景 + 1 个人际延伸场景。每个场景：

- **背景**（跟用户实际情况贴近，如果用户在聊 React 就举 React 例子）
- ❌ 反面示范（从用户自己的消息改写）
- 🔍 问题分析
- ✅ 正面示范（应用推荐框架）

**灵感库**：[`references/scenarios-and-exercises.md`](references/scenarios-and-exercises.md) 有 5 个示例场景——**用作结构参考，不是直接输出**。

### 练习选择

从 4 个练习库里挑 **1-2 个最对口**的（不要 4 个全推）：

| 用户主问题 | 推荐练习 |
|---|---|
| 角色/约束/输出低分 | 练习 1（RACE 速写） |
| 多维度分散 | 练习 2（8 维度清单） |
| 反馈/元沟通低分 | 练习 3（反馈循环） |
| 第 2+ 次使用 / 想长期提升 | 练习 4（LLM-as-a-Judge） |

第 3+ 次使用的用户：推新练习，别重复上次推过的。

详细模板 → [`references/scenarios-and-exercises.md`](references/scenarios-and-exercises.md)。

## 简洁模式

用户说 "简洁版 / 太长了 / 只要重点"，调整输出：

- Phase 1：只输出 Top 3
- Phase 2：只推 1-2 框架（仍要搜索验证）
- Phase 3：只给 1 个 AI 场景 + 1 个练习
- Phase 4：正常持久化（不缩短）

---

## 错误处理

| 情况 | 处理 |
|---|---|
| 会话 < 3 轮 | 输出"对话太短无法分析，建议再聊几轮或直接告诉我想优化哪方面"，终止 |
| Web 搜索失败 / 无结果 | 回退到书籍/权威引用：CO-STAR → GovTech 文档；RACE → McKinsey；SCQA → 《金字塔原理》Minto；NVC → 《非暴力沟通》Rosenberg；CoT → 《Thinking Fast and Slow》Kahneman。**并在报告里标注"未能搜索到在线来源，已回退书目引用"** |
| 用户否认某个打分 | 不要硬杠。回复"这个维度你比我更了解自己，我把它从 Top 问题里拿掉，改推 [下一位]" |
| 用户拒绝某个框架建议 | 问一句"是觉得场景不匹配还是框架本身不喜欢？"，根据回答换框架或调整场景 |
| 历史数据损坏（`index.json` 解析失败） | 备份为 `index.json.bak` → 重建 → 标注警告 |

## Common Mistakes

| 错误 | 修正 |
|---|---|
| 引用"用户消息"但其实是编造的 demo 文字 | 必须逐字引用真实消息 + 标注 turn 编号 |
| 推荐框架没搜索 | 强制至少 2 个在线来源 + 输出 URL |
| 场景用写死的 React/K8s 模板 | 从用户真实对话生成，贴近用户当前上下文 |
| 4 个练习全推 | 按用户主问题挑 1-2 个 |
| 输出完忘了持久化 | Phase 4 必做，不做等于下次 Phase 0 拿不到历史 |
| 创建 `data/sessions/xxx.json` 嵌套目录 | 错。扁平 Markdown 报告 + flat `frameworks/` 子目录，按 [persistence.md](references/persistence.md) 规范 |
| 说"Phase 0 自动执行"就真以为会自动 | Claude 按顺序执行——Phase 0 是**第一步**，必须主动读 `index.json` |
