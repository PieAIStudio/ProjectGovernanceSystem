# knowledge-node

## 这个技能是干嘛的

当用户在学习代码、研究项目、追问"为什么这样设计"、或者要把已有对话的知识沉淀到 Obsidian vault 时，这是主力技能。

**三条核心原则：**
1. **知识要落地** —— 但由用户决定**什么值得落地**，不自动强存
2. **图优于字** —— Mermaid 图建立大脑模型比文字快，每篇至少一张
3. **原子 + 网状** —— 每篇只讲一件事，靠 `[[双链]]` 编织成网

**宁缺毋滥**：低价值内容存进 vault 比不存更糟——它会稀释真正稀缺的节点，让后期检索信噪比下降。遇到以下情况主动提醒用户"信号较低，建议跳过"：
- 纯机械操作（改版本号、删一行）
- 通用常识（npm install 干嘛用的）
- 项目特定一次性信息（临时变量名、具体行号）
- 用户在这次对话里其实没学到新东西的内容

## Step 0 — 解析上下文（三阶段共用，必须先做）

- `SOURCE_PROJECT` = 用户正在学习/分析的外部项目或资料来源
- `KNOWLEDGE_BASE` = Obsidian vault，路径：
  `/Users/yuanfei/Library/CloudStorage/OneDrive-Personal/Obsidian/PieVaultLocal`

**判断规则：**
1. 用户给了明确外部路径 → 那就是 SOURCE_PROJECT
2. 没给外部路径 → 用当前工作目录作为 SOURCE_PROJECT
3. vault 本身不是默认的 SOURCE_PROJECT，除非用户就是在研究 vault 本体

**为什么要先做**：三阶段都会涉及"从哪里取代码"和"存到哪里"，两个角色混淆后文档会完全错位（比如把 vault 里的笔记当教材源，绕成无限循环）。

## 工具选择

| 场景 | 用什么 | 为什么 |
|---|---|---|
| 写学习文档 | `Write` / `Edit` 直接写文件系统路径 | 保证输出是纯 Markdown，从 `# 标题` 开始，不混入 YAML frontmatter |
| 查找 / 搜索 vault 已有笔记 | `mcp-obsidian` 的 search / list / get_file_contents | 原生理解 wikilink 和 Obsidian 结构，比 grep 文件系统更准 |
| 读源项目代码 | Read / Grep / Glob | 标准文件系统工具 |

**关于 YAML frontmatter**：学习文档从 `# 标题` 直接开始，不写 `---` header、不写 `type/tags/up` 这类 properties。

为什么：这个 vault 的知识网络 SSOT 是**正文里的 `[[双链]]` + `## 延伸阅读`**。frontmatter 会造成**二元信息源**——properties 和正文同时描述关系，导致后期维护时两处都要改、两处还经常不一致，反而稀释网状结构的清晰度。

## 意图分流 — 三个阶段

```
用户请求
    │
    ├─ 教学触发词：为什么/是什么/解释一下/帮我学习/教我/帮我研究/这个概念/底层原理
    │   → Phase 1: Teach（只对话输出，不碰文件）
    │
    ├─ 落盘触发词：记一下/存下来/落盘/存成文档/学到了/记录这个知识点
    │   → Phase 2: Save（写文档 + 编织双链）
    │
    └─ 批量触发词：把这段对话里有价值的都存一下/批量整理/扫描这段对话/整理今天学的
        → Phase 3: Batch Scan（扫描全会话 → 评分 → 汇报 → 等确认 → 批量 Phase 2）
```

**特殊情况**：用户一句话里同时出现"解释 + 记一下" → 直接进 Phase 2，产出的文档里已含教学内容。

## Phase 1：Teach（教学模式）

只输出到对话，**不写任何文件**。这个阶段的价值是快速在用户大脑里搭模型——强行落盘反而打断学习节奏，而且有些内容听完就够了不值得留。

### 1.1 锚定真实代码

在 `SOURCE_PROJECT` 里找 1–3 段直接相关的代码，记录**精确文件路径 + 行号**。

**为什么不能跳过**：解释如果漂在抽象层（"一般来说状态管理是..."），知识不会在用户大脑里沉淀。锚到真实代码后，用户能建立"抽象概念 ↔ 我项目里具体这几行"的映射——这才是真正的学习。

**找不到的兜底**：如果源项目里确实没用到这个东西，或者是纯理论问题，直接说"未在当前项目找到对应代码，以下用通用例子解释"。**绝不硬编代码假装是项目里的**——编造的代码片段会污染用户对项目的心智模型。

### 1.2 输出结构

按下面这个结构输出到对话：

---

**日常生活类比**

[用初中生能听懂的类比。必须具体——"就像食谱里写'加盐少许'，而不同厨师理解的'少许'不一样"；不能空洞——"就像一种抽象的规则"]

**在 {ProjectName} 里它是怎么用的**

[从 1.1 找到的真实代码出发解释，粘贴片段并标明来源路径]

```language
// 来自 src/path/to/file.ts:42
code here
```

**核心原理图**

```mermaid
[按概念类型选图种，见 Phase 2 的"Mermaid 图种选择"]
```

### 1.3 结尾固定提示

> 💡 觉得值得留下？说 **"记一下"** 或 **"存下来"**，我生成知识文档。

## Phase 2：Save（落盘模式）

走完整 vault 存盘流程。如果对话里已有 Phase 1 教学内容，**直接基于那个内容扩写成文档**，不要重新推导一遍——否则浪费 token 且两个版本容易不一致。

### 2.0 宁缺毋滥检查

落盘前快速判断这条知识的信号强度：

| 信号 | 判断 | 处理 |
|---|---|---|
| 高 | 有决策点 / 踩坑经验 / 非显而易见的原理 | 直接继续，不打扰 |
| 中 | 常识级但用户花了力气才理解 | 继续，内容可精简 |
| 低 | 机械操作 / 通用常识 / 一次性信息 | 告知"信号较低，不建议落盘"，等用户确认 |

用户明确坚持要存 → 尊重用户，继续执行。这是**他的 vault**，最终决定权在他，技能只负责提醒。

### 2.1 识别项目名

```bash
git -C "$SOURCE_PROJECT" remote get-url origin 2>/dev/null | sed 's/.*\///; s/\.git//'
# 备用：basename "$SOURCE_PROJECT"
```

**命名规则**：保留原始大小写（`PieFlow` → `PieFlow`）；全小写项目名转 PascalCase（`my-app` → `MyApp`）。

### 2.2 目标目录

`{KNOWLEDGE_BASE}/AI-Learning/LearnFromProjects-{ProjectName}/`

- 不存在 → `mkdir -p`
- 已存在 → 用 `mcp-obsidian` 或 `ls` 列出当前文件，**让自己知道这块知识图谱现在长什么样**（后面 2.3 查重和 2.5 编织双链都需要这个上下文）

### 2.3 查重

在目标目录搜主题关键词：
- **有重叠** → 读现有文档，判断是**扩充**（在已有文档里加一节）还是**拆分**（建一个更聚焦的子文档）。原子原则：宁可拆多篇，也不要把一个文档变成大杂烩
- **无重叠** → 进 2.4

### 2.4 选前缀 + 写文档

**前缀优先级（从高到低）**：
`gotcha-` > `versus-` > `debug-` > `concept-` > `workflow-` > `config-`

**为什么这个顺序**：踩坑经验和对比决策在互联网上最稀缺（别人的博客很少写，大多写的是 happy path），而纯概念解释哪里都有。稀缺前缀优先能让 vault 积累**独特价值**，而不是变成"维基百科本地镜像"。写的时候先问一遍"这个知识能归到 gotcha/versus 吗？" 如果能，就不要图省事用 concept-。

| 前缀 | 适用场景 |
|---|---|
| `gotcha-` | 反直觉设计、容易踩的坑、陷阱 |
| `versus-` | 两个方案的差异与选择理由 |
| `debug-` | 调试思路与排查路径 |
| `concept-` | 某个概念/技术/函数是什么 |
| `workflow-` | 一个过程/流程如何运转 |
| `config-` | 配置相关的解释 |

**文档结构**：

```markdown
# {prefix}-{kebab-case-name} 中文标题

> 一句话副标题：点明核心

---

## 日常生活类比

[类比解释]

## 在 {ProjectName} 里它是怎么用的

[真实代码 + 路径]

```language
// 来自 src/xxx.ts:42
code here
```

## 核心原理图

[至少一张 Mermaid 图，类型按下方选]

---

## 延伸阅读
- [[related-doc-A]]
- [[related-doc-B]]
```

**Mermaid 图种选择**：

| 内容类型 | 首选图种 | 例子 |
|---|---|---|
| 流程 / 过程 | `flowchart` | 请求处理流程 |
| 角色交互 / 异步消息 | `sequenceDiagram` | 前后端握手 |
| 生命周期 / 状态转移 | `stateDiagram-v2` | 订单状态机 |
| 层次 / 架构 | `flowchart` 或 `classDiagram` | 模块依赖 |
| 对比 / 决策分支 | `flowchart` with 分支 | versus 类文档 |

**复杂话题可以画多张**（比如 `workflow-` 常需要 flowchart 讲主流程 + sequence 讲关键交互）。

**为什么强调图**：文字是线性的，图是二维的，大脑处理结构化信息时图快好几倍——尤其 Obsidian 原生渲染 Mermaid，画图 ROI 极高。节点名必须说出具体东西（"用户点击提交" / "Zustand store 更新"），不能写"开始 → 处理 → 结束"这种空壳。

### 2.5 编织双链（整个技能最关键的一步）

这一步决定了 vault 是"文档堆"还是"知识网"。不要只机械加 1-2 个链接。

**加双链的启发式**（从高到低）：

1. **同 `SOURCE_PROJECT` 下的同话题文档** —— 几乎必链，它们是兄弟节点
2. **跨类型互补关系** —— 优先链。比如 `gotcha-xxx` 一定要双向链到对应的 `concept-xxx`：踩坑经验是概念的反面教材，用户遇到坑时会回查概念，读概念时也该看到坑
3. **同类型平行关系**（两个 `concept-` 讲不同东西）—— 只在**语义真正相关**时才链，不要"看起来相关就连"
4. **不加反链的情况** —— 老文档已经指向该主题、或关联很弱——强行加反而稀释图谱信噪比

**执行步骤**：
1. 扫描目标目录（优先 `mcp-obsidian` search）找候选文档
2. 在新文档的 `## 延伸阅读` 加出链（通常 2-4 条）
3. 对**关系强**的老文档（按上面启发式筛，通常 1-3 篇），用 `Read` + `Edit` 补反链到它们的 `## 延伸阅读`

## Phase 3：Batch Scan（批量扫描）

扫描当前完整会话，找出值得落盘的话题，**汇报 → 等确认 → 批量落盘**。绝不自动动手创建，因为用户对"什么值得留"有最终判断权。

### 3.1 扫描对话识别候选

按**话题块**扫描会话——一个话题块 = 围绕同一技术主题的连续 turn（用户提问 + AI 回答 + 可能的追问）。话题切换的标志：用户提了个跟上一块明显无关的问题、或者用户明确说"换个话题"。

列出所有出现过的独立知识点、概念、踩坑。

### 3.2 二维信号评分

每个候选独立打分。两个维度：

**维度 A：内容绝对稀缺度**

| 分 | 含义 |
|---|---|
| 1 | 纯机械操作 |
| 2 | 操作 + 少量上下文 |
| 3 | 通用常识级知识 |
| 4 | 有决策点/踩坑/非显而易见的原理 |
| 5 | 深度洞见/架构决策 |

**维度 B：用户实际学习深度**

| 分 | 含义 |
|---|---|
| 1 | 用户秒懂，几乎没互动 |
| 2 | 简单确认 |
| 3 | 有一两轮追问 |
| 4 | 多轮追问 / 用户有"原来如此"的反应 / 明显花力气才理解 |
| 5 | 用户主动延伸、联想、应用到别的场景 |

**总分 = max(A, B)** ——不是平均。

**为什么取 max**：用户花 20 分钟踩出来的"常识"（维度 A=3, 维度 B=4），对这个用户来说就是宝贵知识。反过来，一个客观很深的架构洞见，如果用户只是瞄了一眼没消化（A=5, B=1），现在落盘也没意义——以后回看只是陌生文字。max 让评分同时尊重**知识本身**和**这次对话里发生的学习**。

**处理规则**：
- 总分 ≤ 2 → 跳过，报告里注明
- 总分 = 3 → 默认跳过，列在"待定"让用户决定
- 总分 ≥ 4 → 纳入创建列表

### 3.3 汇报 + 等确认

```
📋 批量扫描结果（当前会话）

✅ 准备创建（信号 ≥ 4）：
  1. [话题 A]（A=5/B=3→5分）— gotcha-xxx，具体描述...
  2. [话题 B]（A=3/B=4→4分）— concept-xxx，用户花力气理解的常识...

⚠️ 跳过（信号 ≤ 2）：
  - [操作 C]（A=1/B=1）— 纯机械操作

🟡 待你决定（信号 = 3）：
  - [话题 D]（A=3/B=3）— 边缘，你要吗？

确认开始？还是要调整列表？
```

**必须等用户明确确认再开始创建**——不要抢跑。

### 3.4 批量执行 Phase 2

对每个选中话题独立走完 Phase 2（2.0→2.5）。2.0 质量门禁因为已评分，直接通过。

每完成一篇输出进度：`✅ [1/3] concept-xxx 已保存`
