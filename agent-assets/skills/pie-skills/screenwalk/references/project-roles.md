# ScreenWalk Project Roles

Use this reference when the default run is not enough and the project needs specialized lenses.

Do not replace `personas.md`. That file is the optional challenge panel. This file is for project-specific role adapters.

## Role selection

Pick only roles that change the evaluation.

Good roles:

- notice different problems
- protect a real project goal
- map to an intended audience or reviewer
- clarify commercial, design, game-feel, trust, or doctrine tradeoffs

Bad roles:

- duplicate the same generic user
- flatter the owner's preference
- create fake consensus
- make the report huge without changing decisions

## Common app roles

| Role | Use when | What it protects |
|---|---|---|
| First-time user | Onboarding, comprehension, first run | Basic clarity and next-step confidence |
| Returning user | Core loop, repeat use, dashboard | Efficiency and habit formation |
| Skeptical payer | Pricing, upgrade, quota, checkout | Trust and value exchange |
| UIUX designer | Layout, hierarchy, consistency, interaction | Visual and interaction craft |
| Doctrine / brand reviewer | Mission-heavy projects | Whether the interface violates higher principles |

## Common game roles

| Role | Use when | What it protects |
|---|---|---|
| First-time player | Onboarding, first match, first battle | Can the player understand and start playing? |
| Game-feel reviewer | Combat, rewards, animation, feedback loops | Feedback, rhythm, readability, delight |
| Rules explainer | Turns, victory, resources, PvE/PvP rules | Whether the player understands what happened and why |
| Economy / payer reviewer | Store, upgrades, gacha, quotas, premium prompts | Fair value exchange and non-disgusting monetization |
| UIUX designer | HUD, overlays, layout, screen hierarchy | Visual coherence and interaction clarity |
| Game art director | Character staging, readable silhouettes, visual composition | Character readability, style discipline, and obstruction-free key art |
| HUD designer | Battle HUD, cards, resources, target labels, timers | HUD clarity without covering playfield or current action |
| Audio / feedback reviewer | Sound, hit/confirm/error feedback, rhythm | Whether feedback is perceivable and not noisy or absent |
| IP / brand reviewer | Character, world, doctrine, lore, franchise tone | Whether brand/IP meaning lands through play, not lectures |

## Non-Heroes role adapter example

For Non-Heroes, consider:

- `first-time player`: can they understand the loop and start playing?
- `game-feel reviewer`: does battle feedback, timing, reward, and state change feel like a real game?
- `UIUX designer`: are style, hierarchy, spacing, overlays, and component language coherent?
- `game art director`: are character heads, silhouettes, clay-figurine staging, and nameplates readable without obstruction?
- `HUD designer`: does battle state stay readable without covering the battlefield, cards, targets, or next action?
- `skeptical payer`: does monetization feel like value exchange rather than pressure?
- `doctrine / brand reviewer`: does The Swimmer’s Guide to the AI Tsunami appear as meaningful product direction rather than sermon text?

Use at most 3 to 5 roles per run unless the user explicitly asks for exhaustive review.
