## 0. 一句话定位

一个**诊断工具**:模拟特定读者去读已写章节,产出一份"读者体验图谱"——把读者在每一章 / 每一拍的**情绪起伏**和**认知变化**画出来,并指出哪里塌了、为什么、怎么改。

类比:它是作者请来的"第一个读者",而且这个读者会把自己每一拍的反应如实记录、汇成一张图。

---

## 1. 目标与范围

### 1.1 要做的

- 读已写章节,模拟读者反应,产出可视化诊断报告。
- **唯一用途:诊断改稿**——给作者指出问题章节 / 节拍 + 原因 + 修改方向。

### 1.2 明确不做的(非目标)

- ❌ 不接入写作回路(不喂 `/继续写`)。
- ❌ 不做"趋势体检"式的周期健康报告。
- ❌ 不替作者改稿——只诊断、只给方向,改不改、怎么改由作者定。

> 以上是澄清阶段 Q1 明确排除的。日后若需要,可另立技能,不在本技能范围内。

### 1.3 必须守住的设计铁律(即"四个坑",由 AI 总结、用户认同)

| 铁律 | 说明 | 对策落在 |
|---|---|---|
| **读者认知 ≠ 角色认知** | 技能算的是**读者**脑子里的东西,不是角色的。角色在台上痛哭,读者可能在解气、甚至在出戏。两者必须分开。 | §2、§6 数据结构 |
| **防剧透(反上帝视角)** | AI 若一次读完全书再回头评第 3 章,它的"第 3 章感受"已被结局污染。必须按章顺序、读一章冻结一章。 | §5 架构 Approach B |
| **颗粒度到"节拍"** | 一章里情绪是起伏的,按整章取平均会把波动抹平。分析必须细到 beat。 | §6 `beats`、§9 钻取 |
| **可信度** | 模拟读者会产出"听起来很对"却无法证伪的内容。每个判断强制附 `why` + 引文;并有独立验证方案。 | §6 `why`、§10 验证 |

## 2. 核心理念:情绪 + 认知 双层

读者的体验由两条线构成,技能两条都要抓:

- **情绪线**——读者此刻"什么感受":解气、揪心、想笑、憋屈……
- **认知线**——读者此刻"知道了什么、在想什么":获得了什么信息、跟角色之间拉开了什么"信息差"、心里悬着什么问题。

deepfake 的发动机是**戏剧反讽**(读者知道黄心语是卧底,丁一 / 戴尔蒙在鼓里)——它本质是一个"读者认知 > 角色认知"的持续信息差。技能把"戏剧反讽"当一等公民单独跟踪(见 §6)。

> 注:测试用的 steel-monster-v4 同样是戏剧反讽驱动——其 `plot-beats` 规划稿里反复出现"读者提前知道 X、角色不知道"的标注,正好供技能对照。

## 3. 模拟读者:三个

技能同时扮演三个读者,各管一条**不重叠的"失败轴"**。每个读者配一份**判断清单**(否则会演成"AI 脑补的刻板印象")。

### 3.1 核心目标读者 —— 测"爽不爽 / 代入感"

- **来源**:每本书的 `idea_seed.md` 目标读者画像,逐书生成。
- **deepfake 实例**:25–40 岁都市读者;犯罪喜剧 / 黑色幽默爱好者;《疯狂的石头》《无名之辈》《唐探》受众;读网络连载。
- **判断清单**:
  - 爽 ← 聪明的反转、解气的报应、小人物挣扎被理解、黑色幽默精准命中。
  - 弃书 ← 节奏拖、笑点哑火、主角连续犯蠢且无成长。
  - 出戏 ← 巧合太假、人物降智、作者跳出来说教。

### 3.2 挑剔老书虫 —— 测"套路 / 逻辑硬伤"

- **来源**:复用框架 `/双审` 已有的"极严苛老书虫"人设,保持一致。
- **画像**:读过上千本书的老油条,见过所有套路,笑点阈值极高(老梗免疫)。
- **判断清单**:
  - 扣分 ← 似曾相识的桥段、可预测的反转、重复结构(走平)、AI 腔、逻辑硬伤。
  - 加分 ← 真正的新意、扎实的细节、敢于不落俗套。

### 3.3 路人 / 新读者 —— 测"门槛 / 弃书点"

- **画像**:不是类型铁粉,从推荐位点进来,耐心低,1–3 章内决定去留。
- **判断清单**:
  - 弃书点 ← 开头信息过载、人物太多记不住、不知道主线钩子是什么、前 3 章没有抓人点。
  - 留下来 ← 迅速给出清晰钩子、主角讨喜、场景好懂。

> **构建顺序**:先用「核心目标读者」把整套引擎跑通、验证,再接入老书虫和路人(见 §11.1)。

## 4. 情绪模型:五维基底 + 复合情绪

### 4.1 五维基底

读者情绪用一组**基底维度**表示,各 **0–100**。它们是图上的固定曲线,也是合成其它情绪的"原色":

> **悲伤 · 想笑 · 紧张 · 高兴 · 愤怒**

**为什么是这五个**:前四个是作者最看重的维度;"愤怒"是独立的第五根支柱——它无法由其它四个合成(悲伤是"退缩"、愤怒是"进攻",方向相反),而"憋屈"(憋着的怒)、"解气"(放掉的怒)都靠它才拆得全。

### 4.2 复合情绪 = 基底的组合

解气、憋屈、感动等**不是独立维度**,是基底的组合 / 轨迹。技能**识别并标注**它们(标在曲线上的事件点),但它们不占轨道。技能自带一张**"复合情绪查表"**(首版,可在实跑中扩充):

| 复合情绪 | ≈ 基底组合 / 轨迹 |
|---|---|
| 解气 | 高兴↑ + 愤怒↓(积压的愤怒被清算) + 紧张↓;须接在一段低谷之后 |
| 憋屈 | 高兴低 + 悲伤微 + 愤怒被压住(想发作而不能) |
| 爽 | 高兴↑ +(想笑或解气);期待被满足或超额满足 |
| 感动 | 高兴 与 悲伤 同时偏高(温暖的泪点) |
| 义愤 | 愤怒↑,且指向反派的不义 |
| 揪心 / 担忧 | 紧张↑ + 悲伤微 + 高兴低;针对喜爱角色的处境 |
| 尴尬 / 出戏 | 想笑↓ + 轻微紧张;角色行为让读者替其难堪 |
| 爽快 / 痛快 | 解气的强化版 |

> "解气"等**轨迹型**复合情绪需要"前情低谷"才能判断 → 依赖 `reader_state` 里的 `recent_arc` 字段(见 §6.1)。罕见、查表外的情绪,仍可用开放文字 `named.name` 兜底,一个不丢。

### 4.3 认知层字段

- `new_info` —— 本拍读者获得的新信息。
- `info_gap` —— 信息差的开 / 合。
- `dramatic_irony` —— 戏剧反讽(读者知、角色不知)的持续跨度。
- `open_questions` —— 读者悬着的问题,分**悬念**(担心接下来)/ **好奇**(想知道过去的真相)。

## 5. 架构与落点

### 5.1 Approach B:分章隔离子 Agent

- 技能按 `SKILL.md` 的 SOP 驱动一个**逐章循环**。
- 每一章,派一个**全新子 Agent**——它是**专用子 Agent 类型,不配读取原稿的权限**。把"本章正文 + 读者状态 + 三个读者人设 + 前 1–2 章原文 + 复合情绪查表 + 分析指令"**全部 inline 塞进它的指令**。→ 防剧透**双重锁死**:① 内容 inline 给,不给原稿路径;② 工具层就没有翻后文的能力。
- 子 Agent **自己把分析写盘**(写进 §5.2 产物目录),只向主技能返回**一句极简摘要**(如"第 7 章完成,2 拍标问题,state_health=ok")。→ 主技能上下文**不随章数膨胀**,几百章也不会爆。
- 主技能全程不读章节正文、不持有完整分析 → 主技能自己也不会攒下剧透。
- 分工:**确定性的活**(校验 JSON、滚动状态、渲染 HTML)由 **Python helper** 做;**语义判断与写分析**由**子 Agent** 做。

## 6. 数据结构

整个技能靠两份 JSON 运转。

### 6.1 `reader_state.json` —— 三个读者读到此刻的"记忆"

滚动前进,也是防剧透的载体:每章子 Agent 只拿到它 + 当前章。

```jsonc
{
  "novel": "steel-monster-v4",
  "last_chapter": 7,                       // 处理到第几章(断点续传)
  "updated_at": "2026-05-21T10:00:00",

  "shared": {                              // ↓ 三读者共享(读的是同一批字)
    "facts_known": [
      { "id": "f001", "since": 1, "fact": "读者已掌握的剧情事实" }
    ],
    "dramatic_irony": [
      { "id": "di01", "since": 2, "reader_knows": "黄心语是卧底",
        "in_dark": ["丁一", "戴尔"], "status": "active" }
    ],
    "open_questions": [
      { "id": "q01", "since": 3, "q": "丁一逃得出冥币厂吗",
        "type": "悬念", "status": "open", "resolved_at": null }
    ]
  },

  "readers": {                             // ↓ 每个读者各一份
    "core": {
      "engagement": 78,                    // 投入度 0-100
      "emotion_now": {                     // 当前情绪基线(曲线最新一点)
        "basis": { "悲伤": 15, "想笑": 40, "紧张": 50, "高兴": 25, "愤怒": 20 },
        "label": "替主角揪心"
      },
      "recent_arc": "近 3-5 章情绪走势摘要 —— 让子 Agent 判断轨迹型复合情绪(如解气需接低谷)",
      "dispositions": {                    // 对角色的态度 = 情绪的依据
        "丁一":   { "score": 35,  "why": "贪,但帮老太太那下让人心软" },
        "胡卫南": { "score": -70, "why": "印冥币还设局害人" }
      },
      "fatigue": []                        // 疲劳信号(老书虫/路人更易累积)
    },
    "veteran":  { "...": "同核心结构 —— 更挑剔、疲劳更快" },
    "newcomer": { "...": "同核心结构 —— 更易看不懂、更早想弃书" }
  }
}
```

### 6.2 `ch####.analysis.json` —— 某一章的体验记录

每章子 Agent 读完写盘,按节拍(beat)拆开。

```jsonc
{
  "chapter": 7,
  "title": "封锁边缘",
  "beats": [
    {
      "id": "ch7-b1",
      "summary": "丁一深夜潜回冥币厂打探",
      "cognition": {                                 // 认知层
        "new_info": ["读者确认胡卫南见过戴尔真脸"],
        "info_gap": { "opens": ["丁一会被发现吗"], "closes": [] },
        "dramatic_irony_touch": "无变化"
      },
      "emotion": {                                   // 情绪层 —— 三读者各一份
        "core": {
          "basis": { "悲伤": 25, "想笑": 30, "紧张": 80, "高兴": 10, "愤怒": 15 },
          "named": [ { "name": "紧张", "why": "怕丁一被抓 —— 引:『脚步声越来越近』" } ]
        },
        "veteran":  { "...": "同核心结构" },
        "newcomer": { "...": "同核心结构" }
      },
      "flags": []                                    // 这一拍无诊断问题
    },
    {
      "id": "ch7-b2",
      "summary": "丁一和戴尔第三次互相指控",
      "cognition": { "new_info": [], "info_gap": { "opens": [], "closes": [] } },
      "emotion": { "...": "..." },
      "flags": [
        { "type": "走平·套路", "hit": ["veteran", "newcomer"],
          "why": "互相指控来回第 3 轮、信息零增量;想笑 52→12",
          "fix": "换成被迫合作的反差,或并掉一轮" }
      ]
    }
  ],
  "chapter_rollup": {
    "shape": "先紧后平",
    "end_hook": { "level": "中", "what": "丁一捡到一个 U 盘" },
    "verdict": "可用;第 2 拍节奏拖,建议处理"
  },
  "state_update": {                                  // 交给 helper 滚动 reader_state
    "add_facts": ["..."],
    "add_questions": ["..."],
    "resolve_questions": ["q03"],
    "disposition_shifts": { "core": { "丁一": "+5 有胆识" } }
  },
  "state_health": "ok"                               // ok | warn —— 状态健康度
}
```

**量纲约定**:所有 `basis` 维度、`engagement` 均 0–100;`disposition.score` 为 -100~+100。量纲在指令中写死,避免不同次运行理解不一。

## 7. 端到端流程

跑一次 `/读者图谱 <小说>`:

**① 准备**
- 读 `idea_seed.md` → 生成"核心读者"人设;载入"老书虫""路人"人设模板。
- 找 `reader_state.json`:有 → 断点续传;无 → 新建空状态。
- 对比"已写章节"与"已分析章节" → 得出待处理清单;检查章节连号。

**② 逐章循环**(严格按章序,对每个待处理章节 N)
- a. helper 读取第 N 章正文。
- b. 打包输入包(见 §5.1)。
- c. 派全新子 Agent(无原稿读取权限),输入包全部 inline。
- d. 子 Agent 把三读者的 `ch####.analysis.json` + `state_update` **写盘**,向主技能只回极简摘要。
- e. helper 从盘上**校验** JSON(字段、量纲)。坏 → 带原因重派子 Agent,最多 3 次。
- f. helper 把 `state_update` 滚进 `reader_state`(同样过校验)。
- g. helper 存盘新 `reader_state` —— 该章分析 + 新状态落盘,即为**断点**。

**③ 渲染** —— helper 汇总全部 `ch####.analysis.json` → 生成 HTML 多轨仪表盘。

**④ 输出** —— 写入 `<小说目录>/04_editing/reader_map/`。

## 8. 错误处理

| 出什么事 | 怎么办 |
|---|---|
| 子 Agent 写出的 JSON 坏 / 缺字段 | helper 校验时发现,带"上次哪错了"重派,最多 3 次 |
| 重试 3 次仍失败 | 干净停下,报"卡在第 N 章",**已存断点不动**,修完可续 |
| 跑一半中断(崩溃 / 手动停) | 每章存档 → 重跑自动从 `last_chapter` 续上 |
| 章节文件缺号 / 乱序 | 准备阶段检查连号,缺了先警告 |
| 子 Agent 幻觉(一本正经胡说) | 流程防不住 —— 靠两道:① 每个判断强制填 `why` 须引正文;② §10 验证 |
| 长篇 `reader_state` 越滚越大 | 每满 N 章做一次"状态压缩":已结悬念归档、老事实并成梗概 |

**误差累积**:第 ②f 步是危险点 —— 某章状态错了,后续章会一路歪。两道保险:`state_update` 过校验;报告里每章带 `state_health` 标记,异常处用户一眼可见,可手动修该章 state 再续传。

## 9. 输出报告:多轨仪表盘

> 示意 mockup:`Docs/reader-experience-map-dashboard-mockup.html`(可直接浏览器打开)。

形态(澄清阶段 Q4 定:A 骨架 + B 钻取 + C 读者独白):

- **顶部 · 读者切换**:核心 / 老书虫 / 路人。主图**一次只画一个读者**(3×5=15 条线叠一起成面条)。
- **主图 · 5 条情绪轨**:5 个基底各一色,X 轴章节、Y 轴 0–100。
- **复合情绪标注**:不占轨道,以竖标记 + 标签标在图上(如「憋屈」「义愤」「解气」)。
- **诊断标记**:问题章节用 ⚠ + 高亮区标出。
- **认知层**:主图下方的条带 —— 戏剧反讽的跨度、信息差何时开 / 合。
- **钻取详情卡(B)**:点击任意节拍 → 展开该拍的三读者 5 维分数(并排对比)、认知细节、诊断 + 修改方向。
- **读者独白(C)**:详情卡内的第一人称引述,把分数翻译成"人话"。

**已定细节**:主图默认单读者(点 tab 切换,不默认叠加);X 轴按章给总览,节拍只在钻取时展开。

## 10. 验证方案

### 10.1 摆正期待

这是**模拟读者**的工具,本质是主观判断,永远无法"被证明绝对正确"。验证的目标是**确认它够可信、够稳定,值得拿来用**——像确认"一个编辑靠不靠谱",而非数学证明。

### 10.2 五道验证(都在 steel-monster-v4 上做)

| # | 验证 | 验什么 |
|---|---|---|
| ① | 对照作者意图 | 技能跑出的曲线 vs steel-monster 的规划稿 `01_analysis/plot-beats-01〜20.md`(逐章写明意图、节奏弧线如"日常→惊悚→荒诞",并明确标注"读者提前知道"的反讽点)。**对不上不等于技能错**——也可能是意图没落到纸面,由用户拍板。 |
| ② | 抽查"像不像" | 抽 5–10 拍,看情绪分 + `why` 引文撑不撑得起判断。靠用户的读者直觉。 |
| ③ | 跑两遍看稳不稳 | 同章节跑两次,曲线形状 / 标出的问题 / 大情绪点应基本一致。打架 = 太飘,回炉。 |
| ④ | 反剧透自检 | 挑一个"后面才揭晓"的信息点,检查揭晓前几章的分析是否"天真无知"。直接验证防剧透架构。 |
| ⑤ | 埋雷测试 | 故意给某章造个明显毛病(如硬塞重复 → 走平),看技能抓不抓得到。验"召回率"。 |

### 10.3 验证是循环

头几遍必然不完美。看哪偏了 → 调读者人设 / 分析指令 / 复合情绪表 → 再跑。先用核心读者跑通验证,再接老书虫、路人。

**测试对象**:`steel-monster-v4` —— 20 章已写;`01_analysis/plot-beats-*.md` 给出逐章意图(规划到第 55 章),`04_editing/front*_plan_vs_actual_diff.md` 是现成的"计划 vs 实际"对照,验证素材充足。技能本身通用,任何小说都能跑;deepfake 待章节攒够后直接上。
