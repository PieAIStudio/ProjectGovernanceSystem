# AI/Human-Friendly Refactor Methodology

> **状态**：通用方法论
> **读者**：准备做重构的人类与 AI agent
> **范围**：跨项目可复用，不写任何项目专属名词
> **创建时间**：2026-04-14
> **最近复核**：2026-05-04

## 0. 这份文档解决什么问题

AI/Human-friendly 重构的目标，不是让代码看起来更漂亮，而是让未来的正确改动更容易发生。

它同时服务四类读者：

- 人类：需要快速建立稳定的心智模型；
- AI agent：需要从显式上下文推断边界，而不是依赖长期项目记忆；
- 测试和工具：需要尽早发现行为漂移；
- 未来功能开发：不应该继续把新能力堆进同一个大文件。

使用这份文档时，默认目标是**不改变产品行为，只改善结构、边界、可读性和可验证性**。

如果重构过程中发现真实产品问题，不要顺手混修。先把它记录为独立问题，再决定是否进入新的工作块。

## 1. 核心原则

### 1.1 边界先于实现

每个 package、模块、大文件，都应该能回答三个问题：

- 它负责什么？
- 它明确不负责什么？
- 它稳定暴露给外部的 API 是什么？

如果答案只存在于某个人脑子里，而不是代码、README 或接口里，这块代码对人类和 AI 都还不够友好。

### 1.2 行数是审查触发器，不是硬规则

大模型可以读很长的文件，但当一个任务要求同时理解太多无关职责时，可靠性会下降。下面这些阈值只表示“需要停下来检查”，不是“超过就必须拆”。

| 信号 | 审查触发器 | 需要问的问题 |
| --- | ---: | --- |
| 单个源码文件 | 约 1,500 行 | 是否混了多个职责？ |
| 顶层函数 / 组件 | 约 200 行 | 是否可以用命名清晰的 helper 或子组件降低负担？ |
| 单个 package 实现 | 约 10,000 行 | 是否藏了多个子系统？ |
| 单目录直接子文件 | 约 15 个 | 是否已经变成 grab bag？ |
| import 区域 | 约 20 行 | 是否耦合了太多领域？ |
| 近期反复改动 | 多次被同一阶段任务触碰 | 是否正在吸收太多功能变化？ |

超过触发器时，动作是“检查并解释”，不是“机械拆分”。

### 1.3 命名就是架构

优先选择能表达意图的名字：

- `fetchUserProfile` 优于 `getData`
- `isProviderConnected` 优于 `status`
- `buildInvoiceLineItems` 优于 `helper`
- `provider-adapters` 优于 `utils`

谨慎使用 `common`、`utils`、`misc`、`helpers` 这类名字。它们会让人类和 AI 都难以判断“这里到底应该放什么”。

### 1.4 公共 API / 合同要小而清楚

package 和模块应尽量有清晰入口和较小的 export surface。过大的 barrel file、层层 re-export，会让接手者不知道哪些符号稳定、哪些只是内部拼装。

公共 API / 合同不只指代码 export。每个项目都应按自身形态识别稳定合同，例如：HTTP / RPC route、CLI 命令、数据库 schema、配置 schema、事件名、消息名、内容 ID、资产 ID、文件路径约定、snapshot shape、测试 selector、被工具读取的 prompt / template。重构时这些合同要么保持稳定，要么明确标为 intentional change。

公共 API 文档至少要说明：

- 哪些 export 稳定；
- 哪些 export 是 experimental / internal；
- 预期消费者是谁；
- 哪些地方不应该 import 或依赖这个模块 / 合同。

### 1.5 依赖必须能局部推理

循环依赖不只是构建味道问题，它会破坏局部推理。重构前应该先判断一个模块处在依赖图的下层、同层，还是上层。

依赖工具应先 audit，再 gate。不要在还没理解现有图谱时，就把新规则变成阻塞全仓的硬门槛。

### 1.6 行为才是合同

如果重构没有明确计划行为变化，那任何行为变化都应当视为 bug。

重构前应先做到：

- 用测试或可复现实验证据捕捉现有行为；
- 除非计划明确允许，否则保持 public API 稳定；
- 一次只移动一个 seam；
- 每个有意义的小步后跑 targeted check；
- 阶段收口前跑与风险匹配的 broader gate。

### 1.7 文档是边界的一部分

README 和架构说明对 AI 很有帮助，但过期文档比没有文档更危险。

好的做法：

- 文档短而准；
- 链接源码入口，不复制大段 API 表；
- 标明最近复核时间或对应里程碑；
- 可行时加轻量 coverage / drift check；
- 每次重构都说明哪些文档需要同步、哪些文档不应该动。

## 2. 标准重构循环

结构性重构建议按这个循环走：

1. **观察**：测量大小、churn、依赖图、测试覆盖、失败高发路径。
2. **定义目标边界**：说明重构后谁负责什么，谁不负责什么。
3. **准备安全网**：先 targeted tests，再 broader gates。
4. **一次移动一个 seam**：只做行为保持型提取。
5. **验证**：每个 seam 后跑最小但有意义的检查。
6. **更新边界文档**：在代码附近更新 README 或模块说明。
7. **复核文档漂移**：确认没有制造新的平行真相源。
8. **用下一次功能验证重构**：看新结构是否真的让后续改动更容易。

如果重构没有降低未来认知成本、耦合或变更风险，就应该停下来复盘。

## 3. 多工作块重构

大型重构不要靠一个巨大改动完成。更稳的方式是设计一组小工作块。

每个工作块应该只回答一个问题：

- 这个边界为什么值得调整？
- 哪些行为必须保持不变？
- 哪些文件或 package 是主要触点？
- 哪些测试或证据能证明没有漂移？
- 这一步完成后，下一步会不会更容易？

多工作块重构的顺序通常是：

1. 先做只读审计，确认边界和风险；
2. 先收窄公共 API，再移动实现；
3. 先拆状态和领域模型，再拆 UI 组合；
4. 先建立验证门槛，再引入自动化 gate；
5. 每完成一组相关工作块，就做一次收口复盘。

工作块可以很多，但每个都必须有清楚边界。数量多不是问题，问题是每个块都在顺手改一切。

## 4. AI 协作纪律

AI 参与重构时，最大的风险不是“写不出代码”，而是：

- 忘记边界；
- 顺手改变行为；
- 漏掉文档同步；
- 把工具输出误当成架构判断；
- 在旧上下文里继续修已经变化的系统。

建议把 AI 角色分开：

- 观察者：只做审计和证据收集；
- 规划者：决定边界、顺序和工作块；
- 执行者：只按当前工作块实施；
- 复核者：检查行为、文档和边界是否漂移。

同一个 AI 可以在简单任务里承担多个角色，但报告里必须说清楚它在什么时候切换了角色。

## 5. 工具箱

工具应该辅助判断，而不是替代判断。

| 工具 | 适合用途 | 建议姿态 |
| --- | --- | --- |
| `rg` / `git log` / `git diff` | 查使用点、churn、近期改动 | 默认第一轮 |
| `madge` | 快速查循环依赖和依赖图 | 拆分前 audit |
| `dependency-cruiser` | 固化依赖规则 | 规则验证前 warning-only |
| `knip` | 查 JS/TS 未使用文件、依赖、exports | unused-code audit 候选 |
| `ast-grep` | 结构化搜索、lint、codemod | 先窄查询，再考虑 rewrite |
| `biome` | JS/TS lint / format 一体化工具候选 | 仅在项目没有稳定工具链或明确评估迁移时考虑 |
| `oxlint` | 高速 JS/TS lint 候选 | 可作为加速层评估，不应直接替代现有 gate |
| `ts-prune` | 查 TypeScript 未使用 exports | 不建议作为新项目首选；优先评估 Knip |

工具引入本身也应该是一个受控变更。不要在重构早期一次性安装很多工具并设为 blocking。

## 6. 边界说明清单

不要为了套模板给每个小目录创建 README。只有当重构产生或改变了长期稳定的 package、大模块、跨团队边界或高频 AI 入口时，才需要边界说明。

优先更新已有 README、架构文档或模块说明。确实需要新建文档时，保持短而准，只回答：

- 这个边界负责什么；
- 它明确不负责什么；
- 稳定合同 / public API 是什么；
- 主要入口在哪里；
- 应该用什么测试或检查验证它；
- 当前稳定性是 stable / evolving / experimental；
- 最近复核时间或对应里程碑。

## 7. 完成门槛

重构完成至少要满足：

- 旧边界和新边界都说清楚了；
- public behavior 没变，或者行为变化被明确标为 intentional；
- 被移动代码的 targeted tests 通过；
- 与风险匹配的 broader gate 通过；
- 新文档没有复制另一个真相源；
- commit 或计划里仍然看得出回滚路径；
- 下一个接手的人类或 AI 能更快判断“应该在哪里改”。

## 8. 常见反模式

- 只因为文件长就拆，但没有证明更好的边界。
- 创建一堆小文件，但名字不清楚。
- 为了套模板到处创建 Boundary README，反而制造文档噪音。
- 写了一堆 README prose，却没有办法验证它不过期。
- 一次性装很多 guardrail 工具，并在基线没清楚前全部设为阻塞。
- 同一步里既移动代码又改变行为。
- import graph 变漂亮了，但下一个功能还是必须改同一个巨型文件。
- 把 AI-friendly 理解成“多写注释”，而不是清晰命名、边界和测试。
- 把一次审计发现的所有问题都塞进同一个重构。

## 9. 一句话

只有当新结构能让下一次正确改动更容易做、更容易验证、更不容易误伤时，重构才值得做。
